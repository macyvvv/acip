# ORGANIZER_BEHAVIOR_MODEL

## 1. Purpose

本書は、主催者のユーザー行動を、運営責務と心理障壁を含めて定義する正本である。

主催者は、イベントを「立ち上げる人」であると同時に、

- 成立させる
- 停滞を見つける
- 介入する
- 当日を事故なく回す
- 集金する
- 演奏順とタイムテーブルを整える
- 打ち上げを開催する
- 動画撮影などの記録を管理する
- 次回につなげる

役割を持つ。

そのため本書は、主催者の行動を機能単位ではなく、運営の実務単位で書き起こす。

**役割関係(ADR-0049):** 主催者は参加者の全権利（会の探索・参加登録・曲エントリー・
演奏・自分の担当管理・緊急連絡）を保持したままの**スーパーセット**であり、本書に
記述する運営実務はその上に追加される。主催者は演奏参加できる。本書は主催者固有の
追加行動のみを記述し、`PARTICIPANT_BEHAVIOR_MODEL.md` の参加者行動は主催者にも
そのまま適用される前提で省略する。

## 2. Design View

主催者の行動は、次の5段階で整理する。

1. Discover: 開催設計の入口を作る
2. Entry: 募集を開始し、初動を作る
3. Build: 未成立を減らし、成立へ寄せる
4. Run: 当日運営を成立させる
5. Review: 改善と再開催判断を行う

## 3. Core Interpretation

主催者は「案内を出す人」ではなく、

- 判断材料を揃える人
- 参加者の不安を下げる人
- 停滞を早く見つける人
- 当日の破綻を抑える人
- 次回の再現性を上げる人

である。

したがって主催者向け設計は、募集文の整備だけでは不十分で、
進捗監視、介入、復旧、改善まで含む必要がある。

## 4. Organizer Behavior Model

### 4.1 Discover

#### 行動 - Discover

1. 開催のテーマを決める
2. 開催規模を決める
3. 想定人数と会場条件を決める
4. 募集に必要な前提情報を揃える
5. 開催が現実的かを判断する
6. 演奏候補曲の提案を参加者からも受け付ける設計にする

#### 心理障壁 - Discover

1. どこまで準備すればよいか分からない不安
2. 失敗したら人を集められないという懸念
3. 設計条件が曖昧で決めきれない迷い
4. 過去回との違いをどう出すかの悩み

#### 責務 - Discover

1. 開催条件を明示する
2. 判断に必要な前提を揃える
3. 参加者が不安にならない設計を作る
4. 募集前に抜け漏れをなくす
5. 演奏候補曲の提案権を主催者専有にしない

### 4.2 Entry

#### 行動 - Entry

1. 募集文を作る
2. 曲を登録する
3. 参加導線を整える
4. 初動反応を確認する
5. 反応が弱ければ文面や告知を修正する
6. 参加者からの候補曲提案を受け取り、曲登録へ反映する
7. 開催日前の参加締切、曲別エントリー締切、推薦通知締切を設定する

#### 心理障壁 - Entry

1. 募集文が伝わるか分からない不安
2. 初動が弱いと失敗感が早く来る焦り
3. どの情報を強調すべきか分からない迷い
4. 参加者からの問い合わせに耐えられるかという負荷感

#### 責務 - Entry

1. 必須情報を抜けなく提示する
2. 募集文のテンプレートを維持する
3. 参加者の初動不安を先回りして下げる
4. 反応データを見て修正する
5. 候補曲提案の受付導線を明示する
6. 参加締切後は飛び入り演奏参加を受け付けない前提を明示する

### 4.3 Build

#### 行動 - Build

1. 未成立曲を確認する
2. 不足パートを確認する
3. 停滞している曲を優先する
4. 自動成立支援の候補、除外理由、通知状況を確認する
5. 例外介入順序を決める
6. 成立状態を改善するために曲単位で運営条件を判断する
7. 必要に応じて演奏順やタイムテーブルの仮調整を行う

#### 心理障壁 - Build

1. 未成立が増えると焦りが増す
2. どこに介入すべきか判断が難しい
3. 自動推薦が適切に動いているか分からない不安
4. 例外対応と通常運用の境界が曖昧になる負担感

#### 責務 - Build

1. 未成立の優先順位を示す
2. 介入すべき曲を可視化する
3. 機械推薦の候補、除外理由、通知状況を明確にする
4. 停滞を放置しない
5. 演奏順の仮決定と調整責務を持つ

### 4.4 Run

#### 行動 - Run

1. 当日の進行を確認する
2. 遅刻・欠席・変更を受ける
3. 影響範囲を把握する
4. 代替案を決める
5. 参加者に必要な情報を伝える
6. 現場で進行を維持する
7. 集金を完了する
8. 演奏順/タイムテーブルを確定運用する
9. 打ち上げの段取りを実行する
10. 必要な記録（動画撮影など）を実施・管理する

#### 心理障壁 - Run

1. 当日変更が重なると対応が追いつかない焦り
2. 伝達漏れが事故につながる不安
3. 優先順を誤ると全体が崩れる緊張
4. 現場で判断し続ける負荷感
5. 集金や記録などの付帯責務が進行を圧迫する負担感

#### 責務 - Run

1. 緊急連絡の窓口を一本化する
2. 変更影響を即時に伝える
3. 復旧手順を短く提示する
4. 当日判断を迷わせない
5. 集金、演奏順、打ち上げ、記録の責務を抜けなく実行する

### 4.5 Review

#### 行動 - Review

1. 運営結果を振り返る
2. 何が詰まったかを記録する
3. 次回改善点を決める
4. テンプレートを更新する
5. 継続開催するかを判断する
6. 記録物を整理し、次回への引き継ぎに残す

#### 心理障壁 - Review

1. 疲労で振り返りが後回しになりやすい
2. 改善点が多いと次回に落とし込めない
3. 負荷が高いと継続開催をためらう
4. 良かった点を再利用できない

#### 責務 - Review

1. 改善点を1-3件に絞る
2. 再利用できるテンプレを残す
3. 継続可否を冷静に判断する
4. KPIや進捗を次回に引き継ぐ
5. 記録・会計・タイムテーブルの残務を整理する

## 5. Organizer-Specific Issues

1. 初動48時間の遅れが、そのまま未成立に直結しやすい
2. 未成立曲の救済が後手に回ると、開催全体の成立体験が弱くなる
3. 当日変更の伝達が遅れると、参加者側の不安が大きくなる
4. 振り返りが曖昧だと、テンプレート改善が進まない
5. 継続開催の判断が疲労に引っ張られやすい
6. 集金・タイムテーブル・打ち上げ・記録が暗黙化すると、主催者の責務過多が見えなくなる
7. 主催者が通常時の個別声かけを担う前提にすると、恣意性、連絡漏れ、特定参加者への負荷集中が起きやすい
8. 参加締切と当日受付の境界が曖昧だと、飛び入り参加を許す誤解が生まれ、成立管理と連絡責任が破綻しやすい

## 6. Modern Requirements

1. 初動48時間を監視対象として扱う
2. 未成立曲の優先順位を見える化する
3. 当日変更の影響範囲を即時表示する
4. 振り返りを改善メモとして残す
5. 次回テンプレの更新を手順化する
6. 通常の候補抽出と通知は機械が担い、主催者は会固有ポリシー、例外処理、最終運営責任を担う
7. dry_run、推薦通知エンジンの実行切替、全体停止判断はPlatform Super User権限であり、主催者機能には含めない
8. 参加締切は開催日前に設定し、Run導線を事前登録者の当日運営に限定する
9. 主催者向けナビゲーション/UIは参加者向け導線への加算とし、置き換えない（ADR-0049）

## 7. Relation to Sitemap

本書で定義した主催者固有の追加行動は、以下のページ群に接続される。

1. `/organizer/setup`
2. `/organizer/build`
3. `/organizer/run`
4. `/organizer/review`
5. `/organizer/momentum`
6. `/organizer/rescue`
7. `/run/recovery`
8. `/review/retention`

上記に加え、主催者は参加者としての全ページ（`/participant/discover`,
`/participant/join`, `/songs/entry`, `/participant/build`, `/participant/run`,
`/participant/review`, `/timeline`, `/emergency`）に、他の参加者と同一の権利で
アクセスし続ける（ADR-0049 §2、`SITEMAP.md` §9-9）。主催者向けナビゲーションは
これらを置き換えてはならない。

## 8. Review Rule

1. 主催者向け機能を追加する場合は、本書の行動・心理障壁・責務に必ず紐付ける
2. 本書変更時は `SITEMAP.md` と `USER_VALUE_ANALYTICS_CANON.md` の整合を確認する
