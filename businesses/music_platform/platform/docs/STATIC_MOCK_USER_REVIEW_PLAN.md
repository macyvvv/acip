# STATIC_MOCK_USER_REVIEW_PLAN

## Purpose

`music_platform` 静的プロダクトモックをユーザー確認する際の観測項目、質問、記録形式を定義する。

## Review Scope

確認すること:

1. 参加者が会への参加登録と曲への参加登録の違いを理解できるか。
2. 締切、空きパート、推薦、辞退、緊急連絡の順序が自然か。
3. 主催者が通常声かけではなく、例外処理へ集中する導線に納得できるか。
4. Platform Super User の `dry_run` / 通知実行境界が主催者に漏れていないか。
5. 「遊びの演奏会」「技術審査なし」「推薦は断れる」が安心材料として機能するか。

確認しないこと:

1. API、DB、認証、通知エンジンの実装可否。
2. 本番ルーティング。
3. 法務文言の最終承認。
4. 料金、収益モデル、正式利用規約。

## Review Tasks

### Participant Flow

1. Homeから参加できる会を探す。
2. Event detailで参加締切と空きパートを確認する。
3. 会への参加登録で参加可能パート、通知許可、担当上限、参加可能日程を保存する。
4. 曲一覧から空きスロットを選ぶ。
5. 曲への参加登録を保存する。
6. Myページで担当曲と推薦を確認する。
7. おすすめ参加候補を承諾または辞退する。
8. 遅刻/欠席時の緊急連絡で影響確認を見る。

### Organizer Flow

1. 締切3種と上限を設定する。
2. Dashboardで未成立、推薦状況、例外を見る。
3. Rescueで候補ゼロや特殊編成の推奨順を見る。
4. Runで未到着者と影響確認を見る。

### Platform Super User Flow

1. Platform入口からNotification Gateへ進む。
2. `dry_run` 結果、除外理由、idempotencyを確認する。
3. Platform操作が主催者画面に漏れていないことを確認する。

## Observation Checklist

| Item | Pass signal | Concern signal | Notes |
| --- | --- | --- | --- |
| 会への参加登録 | 参加可能パート等を保存する意味を説明できる | 曲参加と混同する |  |
| 曲への参加登録 | 本人承諾で曲に入ると理解できる | 推薦で勝手に入ると思う |  |
| 締切 | 参加/曲/推薦の締切差を理解できる | 当日飛び入り可能と誤解する |  |
| 推薦辞退 | 断ってよいと理解できる | 断ると評価が下がると感じる |  |
| 緊急連絡 | 早い連絡が復旧しやすいと理解できる | 罪悪感で送信を避けたくなる |  |
| 主催者責務 | 例外処理へ集中すると理解できる | 個別声かけが主催者仕事だと感じる |  |
| Platform権限 | dry_runが主催者権限でないと理解できる | 主催者も送信実行できると思う |  |

## Questions

1. 最初に何をすればよいか迷ったページはどこか。
2. 「保存される内容」と「現在の状態」は区別できたか。
3. 推薦を辞退することに心理的抵抗はあるか。
4. 緊急連絡画面は責められている感じがあるか、それとも復旧のために見えるか。
5. 主催者画面で通常の個別声かけをしない方針は自然か。
6. 現時点で不要または過剰に見えるページ/情報はどれか。
