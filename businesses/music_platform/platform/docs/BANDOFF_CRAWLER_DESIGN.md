# Bandoff Crawler Design

## Goal

Collect the following from bandoff.info:

1. event metadata block at top (バンオフ情報)
2. entry songs section (エントリー楽曲一覧)
3. both tables in entry songs section:
- 成立
- 未成立

Also handle event discovery where event URLs are non-systematic by crawling DOM links from event list pages.

## Verified DOM Anchors

The following anchors were verified from raw HTML on 2026-07-17.

### List page

- URL: http://bandoff.info/event/event_list
- upcoming list container: `#NewEvent-content`
- past list container: `#OldEvent-content`
- pagination container: `.pagination.mb0`
- next-page link: `a[rel="next"]`
- next-page URL pattern: `/event/event_list?page=N&model=NewEvent` (HTML uses `&amp;`, decode before request)

### Event detail page (example: /hitorie02)

- event info block wrapper: `#event-info`
- entry songs block wrapper: `#entry-body`
- complete songs table wrapper: `#completeTableWrap`
- complete songs table class: `.completeTable`
- waiting songs table wrapper: `#waitingTableWrap`
- waiting songs table class: `.waitingTable`
- song detail links pattern inside tables: `/<event_slug>/song/<song_id>`

## Architecture

Two-phase crawler:

1. Discovery crawler (list pages)
- start from `/event/event_list`
- parse links only from `#NewEvent-content` and `#OldEvent-content`
- follow `a[rel="next"]` until absent
- collect unique event URLs

2. Detail extractor (event pages)
- request each event URL
- parse and extract:
  - event info key/value list from `#event-info`
  - complete songs rows from `#completeTableWrap table`
  - waiting songs rows from `#waitingTableWrap table`
- normalize member columns (Vo, Cho, Gt1, Gt2, Ba, Dr, Key)

## Why this works even if URL shape is irregular

Event URLs are discovered from list-page DOM links, not generated by URL rules. This avoids assumptions about slugs and supports custom IDs.

## Parsing Rules

### Discovery page

- only accept links that match `^/[A-Za-z0-9_-]+$`
- reject known non-event roots such as:
  - `/event`
  - `/database`
  - `/guide`
  - `/tag`
  - `/activity`
- keep de-duplicated absolute URLs

### Detail page extraction

1. Event info
- From `#event-info`, parse each label/value row (開催日時, 開催場所, 主催者, 参加費, etc.)

2. Songs
- Parse header row once
- Parse tbody rows into schema:
  - song_title
  - song_url
  - artist
  - status
  - members: {vo, cho, gt1, gt2, ba, dr, key}
  - source_table: complete | waiting

3. Text cleanup
- trim whitespace
- convert full-width spaces to normal spaces where needed
- treat empty cells as null

## Crawl Politeness and Safety

- obey robots.txt by default
- domain lock to bandoff.info
- fixed delay (for example 2-3 seconds/request)
- no JavaScript execution by default (keeps analytics impact lower)
- retry with backoff for transient HTTP errors

## Failure Handling

- list page parse failure: store raw HTML snapshot and URL in failures log
- detail page missing table wrappers:
  - store partial event_info if present
  - mark extraction_status as partial
- schema drift detection:
  - if expected headers are missing, raise warning and dump first table header row for inspection

## Implementation Plan

1. Add bandoff-specific discovery module
- input: event_list URL
- output: unique event URLs

2. Add bandoff detail extractor module
- input: event URL
- output: one event record + multiple song records

3. Add run command
- `crawl-bandoff` mode with parameters:
  - max pages
  - include old events
  - delay
  - output directory

4. Add fixture-based tests
- keep 2-3 saved HTML files for parser regression tests

## Notes on SPA-like behavior

Even though pagination appears interactive, server-rendered HTML still includes `a[rel="next"]` with concrete `href`. Therefore, browser automation is optional for this target; HTTP crawler is enough unless the site changes to client-only rendering.
