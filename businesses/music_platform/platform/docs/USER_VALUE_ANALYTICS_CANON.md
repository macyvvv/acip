## Purpose

`music_platform` における分析の目的を、"数字の最適化" ではなく
"ユーザー価値の最大化" に固定するための正本。

このドキュメントは次を定義する。

- 分析の受益者と提供価値
- 役割分担（DataOps / MLOps / DevOps / Marketing）
- 因果断定を避ける分析ガードレール
- 4セグメントのカスタマージャーニーと心の動き

## Scope

対象: セッション会 / 演奏会 / バンドオフ会 / バンオフ文脈のユーザー体験改善

対象外:

- 売上最大化のみを目的にした分析
- 因果が未検証な状態での断定的レコメンド

## Canonical Principles

1. 成果指標は体験指標を優先する（成立体験、不安低減、継続参加）。
2. `participation_fee` は単独原因として扱わない。
3. 主効果よりも文脈差（会場、主催者、時期、コミュニティ）を先に疑う。
4. 出力は必ず「次の行動」に変換される形で提示する。
5. ドリフトや品質劣化時は Fail Safe（前回正常版継続）を適用する。
6. 演奏技術、上手さ、曲品質を参加資格、推薦条件、成立判定の評価軸にしない。
7. 参加費は視聴品質への対価ではなく、演奏参加の場代、設備費、運営費として扱う。
8. 会への参加登録である EventParticipation と、曲別担当である SongEntry を分離する。
9. 通常の未成立救済候補抽出、順位付け、通知は機械が担い、主催者は例外処理と運営責任を担う。
10. 推薦通知の `dry_run` と実行切替は Platform Super User 権限とし、主催者機能には含めない。
11. 演奏参加は参加締切までに確定した事前登録者のみを対象とし、飛び入り参加は扱わない。
12. 参加締切、曲別エントリー締切、推薦通知締切は別のゲートとして可視化する。

## Role Ownership

### DataOps

- 交絡因子の定義と管理
- 層別/統制設計
- 公開前のデータ品質ゲート

### MLOps

- 係数変動・特徴空間偏りの監視
- モデル Go/No-Go 判定
- 不確実性表示ルールの維持

### DevOps

- 分析出力パイプライン運用
- 監視・アラート・フォールバック
- リリースゲート自動化

### Marketing

- 分析結果の価値翻訳（誰に何を返すか）
- 断定しない表示文設計
- 体験KPIの設計と検証

## Value Targets By Persona

1. 参加者: 自分が演奏できる可能性を高める選択を支援する。
2. 主催者: 未成立曲を減らす次アクションを明確にする。
3. 初参加者: 参加不安を下げ、最初の行動を起こしやすくする。
4. コミュニティ全体: 過度な偏りや疲弊を早期に検知する。

## Analysis Guardrails

- 禁止: 「これをすれば必ず成立する」などの因果断定。
- 禁止: 単一スコアのみ提示して根拠を示さないUI。
- 禁止: 演奏技術、上手さ、曲品質を推定して参加資格や推薦条件に使うUI。
- 禁止: 本人承諾なしに推薦を曲別エントリーへ変換するUI。
- 必須: 最終更新時刻、適用範囲、不確実性の表示。
- 必須: 品質/ドリフトゲート未達時は新規版を公開しない。
- 必須: 機械推薦は候補理由、除外理由、通知状況、ルールバージョンを監査可能にする。

## Four-Segment Customer Journey

4セグメント定義:

1. 参加者-初心者
2. 参加者-リピーター
3. 主催者-初心者
4. 主催者-リピーター

### Stage 1: 認知・参加判断

- 参加者-初心者
  - ToDo: 参加可否を判断、怖くないイベントを探す
  - 心の動き: 興味 > 不安 > 比較 > 安心が欲しい
- 参加者-リピーター
  - ToDo: 条件と相性で参加先を選別
  - 心の動き: 期待 > 効率重視 > 納得
- 主催者-初心者
  - ToDo: 開催イメージを固める
  - 心の動き: 意欲 > 準備負荷への不安
- 主催者-リピーター
  - ToDo: 開催枠とテーマを決定
  - 心の動き: 再現性期待 > 差別化の悩み

### Stage 2: 募集設計・エントリー

- 参加者-初心者
  - ToDo: 空きパート確認、参加登録、1曲目選択
  - 心の動き: 緊張 > 登録完了待ち不安 > 小さな安心
- 参加者-リピーター
  - ToDo: 複数曲の担当最適化、日程調整
  - 心の動き: 判断高速化 > 条件不一致で離脱
- 主催者-初心者
  - ToDo: 募集文作成、曲登録、案内整備
  - 心の動き: 手探り > 抜け漏れ不安
- 主催者-リピーター
  - ToDo: 募集テンプレ最適化、初動を作る
  - 心の動き: 立ち上がり速度への注目

### Stage 3: 成立形成（中盤運営）

- 参加者-初心者
  - ToDo: 未成立曲の状況把握、練習優先順位付け
  - 心の動き: 貢献したい > 迷い > 背中押し待ち
- 参加者-リピーター
  - ToDo: 欠損パート補完、他参加者との調整
  - 心の動き: 貢献意識 > 負荷管理
- 主催者-初心者
  - ToDo: 未成立曲の救済、機械推薦状況確認、情報更新
  - 心の動き: 焦り > 一手不足感
- 主催者-リピーター
  - ToDo: ボトルネック優先対応、告知再配分
  - 心の動き: 冷静な優先順位判断

### Stage 4: 開催直前・当日

- 参加者-初心者
  - ToDo: 持ち物確認、当日動線確認、到着
  - 心の動き: 緊張ピーク > 現地で緩和
- 参加者-リピーター
  - ToDo: 当日進行への適応、演奏集中
  - 心の動き: 実行モード > 満足/不満の即時評価
- 主催者-初心者
  - ToDo: 進行、トラブル対応、コミュニケーション維持
  - 心の動き: 負荷ピーク > 成功時の達成感
- 主催者-リピーター
  - ToDo: 進行最適化、現場判断、ケア
  - 心の動き: 負荷分散 > 成立状態と運営負荷の評価

### Stage 5: 終了後・次回意思決定

- 参加者-初心者
  - ToDo: 振り返り、再参加判断
  - 心の動き: 安堵 > 次も行けるかの自己評価
- 参加者-リピーター
  - ToDo: 次回候補選定、継続参加判断
  - 心の動き: 満足なら継続、不一致なら離脱
- 主催者-初心者
  - ToDo: 反省点整理、次回改善点抽出
  - 心の動き: 疲労 > 学習意欲
- 主催者-リピーター
  - ToDo: KPI確認、運営改善、継続開催判断
  - 心の動き: 継続可否の冷静判断

## Detailed Player ToDo (Canonical)

この節では、各プレイヤーのToDoを「迷いが発生する実行単位」に分解する。

### 参加者-初心者

1. 参加候補を3件以内に絞る
2. 自分が担当可能なパートを1つ選ぶ
3. 参加締切までにEventParticipationを確定する
4. 1曲目のSongEntryを確定する
5. 当日までの準備チェックを完了する
6. 終了後に再参加可否を記録する

### 参加者-リピーター

1. 日程と会場条件で候補を高速フィルタする
2. 複数曲の担当バランスを最適化する
3. 欠損パート補完の優先度を判断する
4. 練習工数と当日負荷を調整する
5. 次回継続参加先を選定する

### 主催者-初心者

1. 開催条件（日時/会場/想定人数）を確定する
2. 募集文と参加導線を作る
3. 曲登録と不足パートの初期設計を行う
4. 未成立曲の救済行動を回す
5. 直前運営チェックを完了する
6. 振り返りから次回改善を1-3件決める

### 主催者-リピーター

1. 過去回との差分で開催設計を更新する
2. 初動（募集開始後48時間）の反応を監視する
3. ボトルネック曲に介入優先順位をつける
4. 事前登録者だけを対象に当日運営を省力化する
5. KPIレビューで次回テンプレを更新する

## ToDo-Support Function Catalog

各機能は「分析表示」ではなく「次の行動を起こせる補助」を目的にする。

### F1 参加候補ショートリスト

- 主対象: 参加者-初心者 / 参加者-リピーター
- 対応ToDo: 候補絞り込み、参加先決定
- 入力: 開催日時、会場、空きパート、進捗
- 出力: 候補3件、選定理由、注意点
- 成功シグナル: 閲覧後24時間以内の問い合わせ/エントリー率上昇

### F2 パート適合ナビ

- 主対象: 参加者-初心者
- 対応ToDo: 参加可能パート登録、1曲目確定
- 入力: 曲ごとの不足パート、参加可能パート、日程、担当上限
- 出力: 参加可能パート候補、曲別エントリー候補
- 成功シグナル: 初参加者のエントリー完了率上昇

### F3 初回連絡アシスト

- 主対象: 参加者-初心者
- 対応ToDo: 初回参加登録
- 入力: イベント情報、参加登録導線、必要項目
- 出力: 登録前チェック、未完了項目、必要時の問い合わせ導線
- 成功シグナル: 参加登録完了率上昇、登録離脱率低下

### F4 準備優先度ボード

- 主対象: 参加者-初心者 / 参加者-リピーター
- 対応ToDo: 練習計画、当日負荷調整
- 入力: 担当曲、残日数、不足パート状況
- 出力: 優先順位（高/中/低）、準備チェックリスト
- 成功シグナル: 直前キャンセル率低下

### F5 募集設計ガイド

- 主対象: 主催者-初心者
- 対応ToDo: 募集文作成、参加導線整備
- 入力: 開催条件、過去テンプレ、不足パート傾向
- 出力: 募集文テンプレ、導線チェック
- 成功シグナル: 募集開始48時間の初回反応率上昇

### F6 未成立ボトルネック可視化

- 主対象: 主催者-初心者 / 主催者-リピーター
- 対応ToDo: 未成立曲救済、介入優先順位決定
- 入力: 未成立曲、欠損パート、更新停滞時間
- 出力: 優先介入リスト、推奨アクション
- 成功シグナル: 未成立曲の放置期間短縮

### F7 直前運営アラート

- 主対象: 主催者-初心者 / 主催者-リピーター
- 対応ToDo: 当日準備完了、トラブル予防
- 入力: 開催まで残時間、未完了タスク、連絡未返信
- 出力: 直前チェック項目、優先順アラート
- 成功シグナル: 当日トラブル件数低下

### F8 次回改善メモ生成

- 主対象: 主催者-初心者 / 主催者-リピーター / 参加者-リピーター
- 対応ToDo: 振り返り、次回改善決定
- 入力: 実績値、未成立要因、行動ログ
- 出力: 次回改善候補1-3件、維持すべき点
- 成功シグナル: 主催者継続開催率と再参加率上昇

### F29 主催振り返りダッシュボード

- 主対象: 主催者-初心者 / 主催者-リピーター
- 対応ToDo: 改善抽出
- 入力: 未成立要因、応答速度、運営負荷
- 出力: 維持点/改善点、次回アクション
- 成功シグナル: 主催継続開催率上昇

### F30 次回候補レコメンド

- 主対象: 参加者-リピーター
- 対応ToDo: 継続参加先選定
- 入力: 過去参加傾向、満足シグナル
- 出力: 次回候補と理由
- 成功シグナル: 継続参加率上昇

### F31 テンプレ自動更新提案

- 主対象: 主催者-リピーター
- 対応ToDo: 運営テンプレ改善
- 入力: 募集文/案内文の成果差分
- 出力: 更新提案（変更点のみ）
- 成功シグナル: 運営準備時間短縮

### F32 コミュニティ健全性レポート

- 主対象: プラットフォーム運営
- 対応ToDo: 偏り・疲弊の早期検知
- 入力: 参加者偏在、主催偏在、再参加率、通知集中率、辞退率、無反応率
- 出力: 健全性指標、注意セグメント
- 成功シグナル: 偏在悪化の早期是正

## Backlog Operating Rule

1. 機能候補は削減前提でなく、まず網羅する。
2. 実装順は Impact（体験改善）x Confidence（再現性）x Effort（開発負荷）で決める。
3. 各機能は必ず「成功シグナル」を持ち、4週間で検証する。
4. 検証不能な機能は採用しない。
5. 因果断定が必要な機能は、実験設計が整うまで保留する。

## Emotion-Aware UX Requirement

各機能は、ToDo達成だけでなく感情遷移を補助する設計を必須とする。

1. 不安が高い段階（初心者の認知/初回連絡）
  - 必須UI: 手順明示、テンプレ、確認済み表示
2. 焦りが高い段階（主催者の中盤運営/直前）
  - 必須UI: 優先順、期限、1クリック導線
3. 判断疲れが高い段階（リピーターの選別/最適化）
  - 必須UI: 候補絞り込み、比較理由、除外根拠

## MVP Implementation Priority

優先順は「体験改善インパクト x 実装負荷」で決める。
ただしMVPは全体台帳の初回着手順であり、機能範囲を5種に限定する意味ではない。

1. F1 参加候補ショートリスト
2. F6 未成立ボトルネック可視化
3. F5 募集設計ガイド
4. F4 準備優先度ボード
5. F7 直前運営アラート

上記5つをMVP第一弾とし、F2/F3/F8は第二弾で導入する。

## Experience KPI (Canonical)

短期:

1. 初回行動転換率（閲覧 -> 問い合わせ/エントリー）
2. 未成立曲の放置期間
3. 主催者の直前救済対応件数

中期:

1. 初参加30日再参加率
2. 主催者継続開催率
3. 不安理由による離脱率
