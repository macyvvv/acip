# 010 Quality Gate

## Conclusion

Every Canonical Asset must pass quality gates before it is treated as repository-canonical.

## Quality Dimensions

| Dimension | Gate Question | Required Result |
|---|---|---|
| Objective Fit | Does it directly serve the Current Objective? | Yes |
| Reusability | Can it be reused without chat context? | Yes |
| Maintainability | Can future maintainers understand and update it? | Yes |
| Robustness | Does it avoid fragile assumptions? | Yes |
| Scope Control | Are scope and out-of-scope explicit? | Yes |
| Risk Control | Are policy, legal, brand, and operational risks noted? | Yes |
| ROI Link | Does it connect to Business, Operational, Learning, or Strategic Value? | Yes |
| Repository Fit | Does it conform to repository conventions? | Yes |

## Review Result

Allowed statuses:

- Draft
- Needs Revision
- Approved
- Rejected
- Superseded

## Minimum Evidence

Each approved asset must include:

- source context
- intended reuse
- quality gate status
- reviewer notes
- version

## Failure Rule

If any required gate fails, the asset remains non-canonical.
