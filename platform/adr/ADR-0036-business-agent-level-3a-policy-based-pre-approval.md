# ADR-0036: Business Agent Platform Level 3a — Policy-Based Pre-Approval of Execution

## Status

Accepted

## Current Design (superseded)

Levels 0-2 (ADR-0032/0033/0034) and Level 3c (ADR-0035) automate proposing, concurrently progressing, and — narrowly, for publishing only — unattended shipping of content, but every generation *execution* decision itself still requires a fresh, scope-specific human approval: `system/core/agent_execution_approval.py::evaluate_business_agent_scope_approval()` must find a matching `approval.json` with `decision_status: approved`, written only by a human via `system/scripts/agent/set_execution_approval.py`. `docs/current/BUSINESS_AGENT_AUTOMATION_READINESS.md`'s Level 3 section names this the one remaining, deliberately unbuilt piece — "reducing/removing human approval," forbidden without a new ADR and the operator's specific, sub-stage-scoped sign-off, per `docs/current/AUTONOMOUS_OPERATIONAL_BASELINE.md`'s standing prohibition and `CLAUDE.md`'s "Human keeps: strategy, approval, capital allocation."

## Proposed Design (adopted)

The operator explicitly asked (2026-07-10, via two rounds of `AskUserQuestion`) to build Level 3a exactly as the readiness doc already envisioned it, choosing "formal introduction as documented Level 3a" over both a narrower one-off exception for two specific pending tasks and a broader, undocumented "trust Claude's real-time discretion with no written artifact" reading. This is the platform's first crossing of the human-approval line.

New components, all additive, built via exploration → design → one adversarial critique pass (the same pattern used for Level 2/3c) that found 9 concrete issues, two serious enough to force a real design change before implementation:

- `system/core/execution_pre_approval_policy.py` + hand-authored, PR-reviewed `system/runtime/agent_handoff/auto_approval_policy.json`: standing, revocable authorization per `(business_id, role_id)` — daily/weekly caps, `authored_by`/`authored_at`/`reason`. Fail-closed (absence, a disabled entry, or a missing file all mean "not authorized"). **Two independent safety checks re-run against the LIVE `agent_role_registry` on every single evaluation, never cached or trusted from the policy file**: the target role's `role_kind` must be `claude_invocation` or `data_fetch` (never `pluggable_provider`), and its `allowed_tools` must contain no mutating tool (`Write`/`Edit`/`MultiEdit`/`Bash`) — both raise loudly, not silently, if violated, since a misconfigured policy is worth surfacing every time it's evaluated.
- `system/core/execution_pre_approval_state.py`: a per-`(business_id, role_id)`-sharded, `file_lock`-guarded state machine (`claimed`/`completed`/`failed` per `task_id`) that is the **single source of truth** for whether a task is currently in flight, already done, or eligible for retry — see Design Decision 2 below for why this shape, not a simpler one, was necessary.
- `system/core/execution_pre_approval_control.py`: a **third, independent** kill switch (`is_pre_approval_paused`/`pause_pre_approval`/`resume_pre_approval`), deliberately not reusing the existing task-proposal switch — see Design Decision 1.
- `system/core/approved_autonomous_execution.py`: a new `_try_policy_pre_approval` method, consulted **only** when `evaluate_business_agent_scope_approval()`'s `reason == "missing_approval"` (i.e., no `approval.json` exists at all for that scope yet) — every other denial reason (rejected, superseded, mismatched, disabled) takes the original, completely unchanged path. **No code anywhere gains the ability to write `decision_status: approved`** — this is a wholly separate authorization path, not a synthesized approval artifact. `docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md` needs zero amendment as a result.
- A policy-executed task now calls `mark_task_status(..., "completed")` itself and persists `authorization_source`/`policy_id` to the runtime execution record and the Approval Console — see Design Decision 3.
- No scheduler, cron, or daemon is added. A human (or an existing manual `run_approved_autonomous_execution.py` invocation) still has to actually run the command; only the *decision* is pre-authorized, not the trigger. That remains Level 3b, not requested.

## Reason for Change

Direct, twice-confirmed operator request (2026-07-11 [session-local date; see chat log for exact turn]): "承認を、自動化して良い。モラルとコンプライアンスの基準は、claudeを信頼する" ("you may automate approval; for moral/compliance standards, trust Claude"), clarified via `AskUserQuestion` into: Claude designs the concrete policy schema and safety checks (the same division of labor already used for Level 3c's disclosure-tag heuristic), but the mechanism itself stays a written, capped, revocable, PR-reviewed policy — never real-time discretionary approval with no artifact. The operator chose "Level 3aとして正式導入(推奨)" (formal introduction as documented Level 3a, the recommended option) over a one-time exception for two specific pending tasks, and over broader undocumented discretion.

## Benefits

- Delivers exactly what was asked: a standing policy that authorizes execution in advance for a scoped, capped, revocable set of `(business_id, role_id)` pairs, without ever touching the actual human-approval-artifact contract.
- The `missing_approval`-only trigger condition means an explicit human rejection, once written, is never overridden by a matching policy — verified by a dedicated negative test (an `approval.json` with `decision_status: rejected` plus a fully matching, enabled, in-cap policy still denies using the original reason).
- Re-validating `role_kind` **and** `allowed_tools` against the live registry on every evaluation (not just at policy-authoring time) means a future PR that adds a mutating tool to an existing `claude_invocation` role, or changes a role's kind, is caught immediately — the safety argument doesn't rest on the policy author having gotten it right once and never having it drift.
- A dedicated kill switch, and a state machine that is the sole source of truth for in-flight/completed/failed status, close a real concurrency bug (see Design Decision 2) before it ever shipped.
- A policy-executed task correctly leaves the Approval Console's candidate list, so a human reviewing the console is never misled into thinking a self-executed task is still awaiting their selection.

## Drawbacks

- **The core, unavoidable tradeoff**: once a human authors and merges a policy entry, every future task in that `(business_id, role_id)` scope executes without a fresh per-task human look at the decision to execute. The human's remaining checkpoints are (a) writing/reviewing the policy itself and (b) `system/scripts/publishing/finalize_content.py`'s existing manual step, if that content later flows into Level 3c. **`finalize_content.py`'s manual step is not enforced by any code to involve actual human review** — nothing stops a trivial wrapper from piping raw generation output through it unread. This is the honest answer to whether Level 3a composed with the already-built Level 3c fully closes an unattended generate→publish loop: it does not, but only because of this one manual, currently unenforced step, not because either design intentionally engineered a stop there.
- **Per-role daily/weekly caps do not bound aggregate compounding across a chain**: an operator authoring policies for multiple roles in the same `next_roles` chain (e.g. `market_research`→`marketing`→`doc_creation`) has no single cap bounding the total pre-approved executions for one business in a day, only each role's own cap. Mirrors Level 3c's own disclosed, accepted "fan-out" limitation.
- **A pre-existing `auto_task_id` collision race in `system/core/business_agent_trigger.py`** (an unlocked `load_queue()` read before `add_task()`/`write_business_agent_handoff()`) becomes more likely and more consequential once pre-approved executions can fire in rapid succession without a human serializing them via the console. Not introduced by this change; a good follow-up candidate.
- **The `missing_approval`-only guard is not filesystem-enforced**: it holds against every in-repo code path today (confirmed — nothing deletes an `approval.json`), but an out-of-band `git` operation dropping a scope's `approval.json` would convert a past rejection into "missing." Documented as an accepted, bounded risk, not oversold as a structural impossibility.
- A third independent kill switch adds to the two already introduced by ADR-0033/0035 — a real, acknowledged cognitive cost for the operator, accepted because the alternative (reusing an existing switch across a genuinely different blast-radius concern, or silently redefining its documented behavior) was worse (see Design Decision 1).

## Impact Scope

- New: `system/core/execution_pre_approval_policy.py`, `system/core/execution_pre_approval_state.py`, `system/core/execution_pre_approval_control.py`, `system/scripts/business_agent/pause_pre_approval.py`, `system/scripts/business_agent/resume_pre_approval.py`, this ADR, and the `system/runtime/agent_handoff/auto_approval_policy.json` / `pre_approval_state/` tree (the latter absent until first used).
- Modified (additive only): `system/core/approved_autonomous_execution.py` (new method + two new result fields with safe defaults), `system/scripts/agent/run_approved_autonomous_execution.py` (prints the two new fields), `app/tools/approval_console_mvp/service.py` (surfaces `authorization_source`/`policy_id`; new `PRE-APPROVAL PAUSED` banner), `docs/current/BUSINESS_AGENT_AUTOMATION_READINESS.md` (Level 3a section + Forbidden Now updates).
- Unaffected: `system/core/agent_execution_approval.py` (read from, never modified — its `missing_approval` vs. other-reason distinction is exactly what this design keys off, unchanged), `system/core/business_agent_automation_control.py` (fully untouched — see Design Decision 1), `docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md` (no amendment needed — see Design Decision 3), `system/scripts/agent/set_execution_approval.py`, all `pluggable_provider` roles, Levels 0-2/3c in full.

## Migration Cost

None. Purely additive; `auto_approval_policy.json` ships absent by default (fail closed everywhere), so this PR changes zero live behavior. The operator adds the first real, conservative pilot entry (e.g. `max_auto_approvals_per_day: 1`) as their own separate, reviewed commit, exactly matching ADR-0035's precedent.

## Recommendation

Recommend, with every drawback above stated plainly rather than minimized. This is the platform's first crossing of the human-approval line, and it is scoped as narrowly as the operator's own explicit choices allow: written policy only (never real-time discretion), read-only roles only (double-checked live, on every evaluation), capped, independently killable, and shipping with zero live policy until the operator explicitly turns one on.

## Rejected Alternatives

- **Real-time discretionary approval with no written policy artifact** (a broader reading of "trust Claude on morals and compliance"): rejected by direct, twice-confirmed operator choice in favor of the documented, standing-policy design.
- **A one-time exception for the two specific pending tasks, no standing mechanism**: rejected by direct operator choice in favor of the formal, reusable Level 3a mechanism.
- **Synthesizing a system-written `approval.json`** (`decision_status: approved`, `approved_by: "policy:<id>"`) instead of a separate authorization path: rejected — this would put a non-human write behind the one field `docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md` explicitly reserves for humans, and would make "was this human-approved or policy-approved" a fragile string convention instead of a structural property of which code path ran.
- **Reusing `is_automation_paused()` instead of a new dedicated switch** (the critique's most significant process-level finding): rejected. That switch's docstring is an explicit, load-bearing promise that pausing it never blocks an explicit human action like manually running `run_approved_autonomous_execution.py`. Having the *same* function behave differently under the same paused state depending on which authorization path a scope happens to use would quietly redefine that existing, correctly-documented contract for one of its two callers.
- **Splitting "already granted a cap slot" from "already executed" across two separately-locked files** (the first draft, and the critique's most serious finding): rejected after the critique traced a genuine race — two near-simultaneous invocations could both pass a separate ground-truth check before either had run, both be told "granted," and both concurrently invoke `BusinessAgentExecutionAdapter.run()`'s genuinely slow (up to 60s) subprocess call, doubling real cost and corrupting an unlocked artifact write. Replaced with a single, sharded, lock-guarded state machine (`claimed`/`completed`/`failed`) that is the sole source of truth for a task's lifecycle.
- **A hardcoded eligible-role frozenset** (mirroring `publishing_policy.py`'s static `allowed_source_roles` set) instead of checking the live registry: rejected in favor of the live check, specifically so a future registry change (a new mutating tool added to an existing role, or a role's kind changed) is caught on the very next evaluation, not only at policy-authoring time.
- **A CLI writer for the policy file**: rejected, mirroring ADR-0035's identical rejection for publishing policy, arguably more so here — this is the platform's first crossing into auto-approved-and-executed generation, and deserves the same PR-review bar as a code change.
