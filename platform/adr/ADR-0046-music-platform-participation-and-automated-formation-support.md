# ADR-0046: Music Platform Participation and Automated Formation Support

## Status

Accepted by operator approval on 2026-07-18.

## Context

`music_platform` は「バンオフ、やろう」が持つ、イベント内で参加者が曲を提案し、
曲ごとのパートへエントリーして演奏曲を成立させる業務モデルを継承しつつ、
モバイルファーストの判断支援、自動成立支援、当日復旧を近代化する。

ここで扱う会は、観客へ演奏品質を提示する発表会ではなく、参加者が演奏するための
演奏会/セッション会である。参加費は視聴品質への対価ではなく、演奏参加のための
場代、設備費、運営費として扱う。したがって、プラットフォームは参加者の演奏技術や
曲の品質を審査、格付け、保証しない。参加者本人がその曲・パートへ参加する意思と
準備して参加したいと思える範囲で、その自己判断を尊重する。

検討の結果、次の問題が明らかになった。

1. 会への参加と曲へのエントリーは別の行動であり、単一の参加操作へ統合できない。
2. 自動成立支援には、会への参加登録時点で参加可能パートが必要である。
3. 曲のパートは単純な在席/空席ではなく、必須/任意、複数人数、担当者を持つ。
4. 主催者による個別の候補探索と声かけは、ルールで処理できる反復作業であり、
   恣意性、連絡漏れ、特定人物への依頼集中を生み得る。
5. 機械推薦は必要だが、本人の承諾なしに曲へ自動エントリーすると、練習可能性、
   人間関係、本人の意思を無視する。
6. 欠席連絡の摩擦を完全に除去すると離脱の外部影響を軽視させる一方、連絡自体を
   難しくすると無断欠席を増やす。連絡前の影響確認という意味のある摩擦が必要である。
7. 飛び入り参加を許すと、成立管理、準備期間、連絡責任、会場運営が破綻しやすい。
   演奏参加は事前登録者に限定し、開催日より前に参加締切を置く必要がある。

## Decision

### 1. Two-stage participation model

参加を次の二段階に分離する。

1. `EventParticipation`: 会へ参加し、その会で対応可能なパートを登録する。
2. `SongEntry`: 参加確定後、特定曲の特定スロットへエントリーする。

演奏参加できるのは、参加締切までに `EventParticipation` が確定した事前登録者のみとする。
飛び入り演奏参加は扱わない。参加締切は開催日より前に設定し、曲別エントリー締切、
推薦通知締切、当日運営開始より上位のゲートとして扱う。

会への参加登録時には、少なくとも次を保持する。

- メインパート（1つ）
- その他の参加可能パート（複数）
- 成立支援通知の受信可否
- 最大担当曲数
- 参加可能日程
- 参加種別（演奏/見学等）
- 参加状態（申請中/確定/待機/辞退等）

「対応レベル」または能力の段階評価は保持しない。セッション会/演奏会は演奏機会の
場であり、演奏技術の審査や曲品質の保証を目的にしない。能力評価は主観的で、
推薦の説明可能性と公平性を悪化させる一方、成立支援に必須ではないためである。
技術不安は本人の辞退理由や任意メモとして扱えるが、参加資格、推薦条件、能力評価
データへ変換しない。

### 2. Canonical participant-part list

標準選択肢は次とする。

| Code | Display name |
| --- | --- |
| `vo` | ボーカル |
| `cho` | コーラス |
| `gt` | ギター |
| `ba` | ベース |
| `dr` | ドラム |
| `key` | キーボード/ピアノ |
| `perc` | パーカッション |
| `sax` | サックス |
| `tp` | トランペット |
| `tb` | トロンボーン |
| `fl` | フルート |
| `cl` | クラリネット |
| `vn` | バイオリン |
| `va` | ビオラ |
| `vc` | チェロ |
| `cb` | コントラバス |
| `manipulator` | マニピュレーター/同期音源 |
| `other` | その他 |

UIでは「基本」「追加」「弦」「その他」へ段階的に分ける。
`other` は自由入力名を併記し、将来の別名正規化を可能にする。

このリストは参加者の参加可能パートであり、曲内の `Gt1`、`Gt2`、
`Acoustic Gt` 等の表示スロットとは分離する。曲スロットは標準コードを参照しつつ、
会・曲固有の表示名を持てる。

### 3. Song-part slot model

曲の各パートは少なくとも次を持つ。

- 標準パートコード
- 曲内表示名
- 必須/任意
- 募集人数
- 現在の担当者
- 受付状態
- エントリー状態
- 譲渡可能状態

曲状態は次の三段階を基本とする。

1. 未成立
2. 必須枠充足
3. 成立確認済み

必須枠充足から成立確認済みへの遷移は、会の設定により自動確定または主催者確認とする。
特殊編成、演奏時間、担当集中、出演順衝突等があるため、全会での自動確定は強制しない。

### 4. Accessible slot representation

曲一覧では色だけに依存せず、次の形を第一の情報担体とする。

- `■`: 充足済み/参加済み
- `△`: 空き枠

`○` は「参加可能」「完全成立」「空き」のいずれにも見えやすく、認知負荷が高いため
使わない。必須枠と任意枠の違いは記号を増やさず、`必須` / `任意` のテキストラベルと
スクリーンリーダー向け文章で示す。各表示には十分なコントラスト、44px以上の操作領域を
用意する。承諾待ち、受付停止等も記号を増やさず状態ラベルを併記する。

### 5. Member visibility boundary

曲一覧では人数と充足状態を優先表示し、曲詳細で各パートの担当者名を表示する。
担当者名から、その本人が公開している参加中の会と、その会での担当曲を確認できる。

プラットフォーム全履歴を身元照会のように公開しない。公開範囲、保持期間、本人設定は
今後のプライバシー仕様で確定する。推薦時には既存メンバーを表示し、本人が人間関係を
含む事情で辞退できるようにするが、非公開の人物ブラックリストはMVPへ導入しない。

### 6. Automated formation support

主催者が候補者を探して個別に声をかける通常運用は採用しない。システムが候補抽出、
順位付け、通知、再通知抑制、監査記録を行う。主催者はポリシー設定と例外処理を担う。

候補者は少なくとも次をすべて満たす。

1. 会への参加が確定済み
2. 必須空きと登録済み参加可能パートが一致
3. 成立支援通知を許可
4. 最大担当曲数および会の曲数上限に余裕がある
5. 参加締切、推薦通知締切、開催までの最低準備期間を満たす
6. 通知上限、辞退後クールダウン、同一曲再通知抑制に抵触しない

順位付けには次を用いる。

1. 成立までの残り必須枠数
2. 開催までの日数
3. 停滞時間
4. 候補者の成立曲数の少なさ
5. 現在の担当負荷の低さ
6. パート一致
7. 過去の通知集中度

システムは根拠と不確実性を添えて通知する。本人が承諾した場合のみSongEntryを作成し、
本人の意思なしに自動エントリーしない。

推薦条件に、演奏技術、上手さ、曲品質の推定値を含めない。推薦は参加可能パート、
必須空き、日程、準備期間、担当負荷、通知許可、通知集中度などの運用条件で行う。
主催者は特殊編成、曲固有の注意事項、進行上の制約を設定できるが、それは会の運営条件
であり、参加者の能力審査や曲品質保証ではない。

辞退理由は能力評価へ変換せず、同種の不適切な推薦を抑制するために使う。

- 練習期間不足
- 曲を知らない
- 担当曲数を増やさない
- 今回はそのパートで参加しない
- スケジュール上難しい
- 当該曲の通知停止

### 7. Fairness, caps, and auditability

希少パートやマルチパート参加者への依頼集中を防ぐため、期間内通知上限、同一曲の
再通知禁止、辞退後クールダウン、同時候補数上限、参加者別集中率監視を設ける。

自動成立支援の `dry_run` はプラットフォーム運営者のSuper User権限として保持する。
主催者向けの通常機能にはしない。主催者が扱うのは会固有の募集枠、上限、成立規則、
準備期間、通知ポリシー、例外処理であり、推薦通知エンジン全体の送信前検証、
実行切替、停止判断はプラットフォーム運営責任に属する。

最低限、次を監査可能にする。

- 推薦日時
- 対象曲/スロット/候補者
- 候補となった根拠
- 候補から除外された理由
- 適用ルールのバージョン
- 通知結果
- 承諾/辞退/無反応
- クールダウン期限

### 8. Event-level constraints

曲スロットとは別に、イベントは参加締切、曲別エントリー締切、推薦通知締切、
総参加人数上限、パート別人数上限、パート別受付停止、見学者上限、キャンセル待ち、
曲数上限、段階的な上限解放日時を持てる。

参加締切後は新規の演奏参加登録を受け付けない。締切後の当日画面は登録済み参加者の
到着、変更、欠席、進行確認を扱うものであり、飛び入り参加受付ではない。

現在の担当負荷は、単純な担当曲数だけでなく、エントリー曲、成立曲、譲渡可能曲、
推薦承諾待ち、会の上限を区別して表示する。

### 9. Absence communication

欠席連絡は到達しやすくするが、即時確定のワンタップ操作にはしない。送信前に、
未成立になる曲、影響するメンバー、代替候補、通知対象を示す。影響確認後に連絡を送り、
無断欠席より有断欠席を促しつつ、離脱の外部影響を不可視化しない。

## Human and Machine Responsibilities

Machine:

- 候補抽出と順位付け
- 根拠付き通知
- 頻度制御と再通知抑制
- 承諾/辞退の記録
- 充足判定と影響計算
- 監査ログ

Participant:

- 参加可能パートと通知許可の登録
- 曲へのエントリー承諾
- 辞退と通知停止
- 欠席時の影響確認と連絡

Organizer:

- 会固有の募集枠、上限、成立規則、準備期間、通知ポリシーの設定
- 候補ゼロ、規約衝突、長期停滞、特殊編成等の例外処理
- 当日復旧と最終運営責任

Platform Super User:

- 推薦通知の `dry_run` 確認
- 推薦通知エンジンの実行切替
- 全体またはイベント単位の停止判断
- 送信前検証、誤通知対応、監査確認

## Consequences

Benefits:

- 主催者の反復的な候補探索と個別連絡を削減する。
- 依頼の恣意性、連絡漏れ、特定人物への集中を制御できる。
- 参加者は推薦根拠、既存メンバー、自分の負荷を見て承諾できる。
- 曲の成立状態を人数、必須性、担当者まで含めて一貫して扱える。

Costs and risks:

- 会、曲、スロット、参加、推薦を分離するデータモデルが必要になる。
- 通知ポリシーの誤設定はスパムや成立率低下を招く。
- 担当者表示と参加履歴にはプライバシー仕様が必要になる。
- 公平性は単一スコアでは保証できず、継続的な監査が必要になる。

## Rejected Alternatives

- 対応レベルを登録する: 主観的で、成立支援に必須でなく、能力評価として誤用されるため却下。
- 会への参加と曲へのエントリーを一操作にする: 参加状態と曲担当が混同されるため却下。
- `Gt1`/`Gt2`を参加者プロフィールへ登録する: 曲固有スロットと標準パートを混同するため却下。
- 主催者が通常時の候補探索と声かけを行う: 機械処理可能な反復判断で、恣意性と漏れを生むため却下。
- システムが候補者を自動エントリーする: 本人の意思、準備可能性、人間関係を無視するため却下。
- 欠席をワンタップで即時確定する: 外部影響を不可視化し、コミットメントを過度に弱めるため却下。
- 人物ブラックリストをMVPへ入れる: センシティブな関係データとなり、必要性と運用境界が未確定のため却下。

## Open Questions

次は未決定であり、本ADRの承認に含めない。

- 各参加履歴の公開範囲、保持期間、本人設定
- `other` の別名辞書と標準コードへの昇格基準
- 会ごとの自動成立/主催者確認の既定値
- 自動通知の具体的な重み、日数、上限、クールダウン値
- 公平性指標とアラート閾値
- 事前決済またはキャンセルポリシー
- 推薦・通知に関する法務文言と異議申立て導線

## Amendment (2026-07-19, see ADR-0049)

`ADR-0049` により本ADRの読み替えと追加が確定した。本ADR自体の決定（two-stage
participation model、canonical part list、song-part slot model、accessible slot
representation、member visibility boundary、automated formation support、
fairness/caps/auditability、event-level constraints、absence communication、
Minimum Data Contracts）は変更しない。以下の2点のみ読み替える。

1. **"Human and Machine Responsibilities" の Participant / Organizer は排他的な
   責任集合ではない。** Organizer は Participant の全権利（会の探索・参加登録・
   曲エントリー・演奏・自分の担当管理・緊急連絡）を保持したまま、本ADRが列挙する
   Organizer 固有の権利（会固有の募集枠/上限/成立規則/準備期間/通知ポリシーの設定、
   例外処理、当日復旧と最終運営責任）を追加で負う。主催者は演奏参加できる。
   Platform Super User は会の参加ロールではなく別軸の運営者ロールであり、本ADRの
   分離（主催者に dry_run/実行切替/停止を与えない）はそのまま維持される。
2. **収益モデルは本ADRの対象外だった。** `ADR-0049` により、プラットフォーム運営者の
   収益源はアフィリエイトのみと確定した。本ADR §Context の「参加費は場代・設備費・
   運営費」はユーザー間費用の性質を述べたものであり、運営者収益の記述ではない。
   アフィリエイト推薦は本ADR §6-7 の推薦ガードレール（根拠+不確実性、断定回避、
   能力評価を条件に使わない）に従い、収益層は成立/公平性層を汚染しない。

詳細は `platform/adr/ADR-0049-music-platform-business-model-and-role-refinement.md`
を参照。
