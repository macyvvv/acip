# BUSINESS_AGENT_AUTOMATION_READINESS

## Summary
Foundation for eventual full automation of the multi-business agent platform (`docs/current/PROJECT.md`'s AI-native-company goal). What's automatic today is the *proposal* of the next task (Level 1/2) and, narrowly, the *execution decision itself* for scopes an operator has explicitly pre-authorized in advance via a standing, capped, revocable policy (Level 3a, below) — every other scope still requires a human to explicitly approve that specific scope, every time, and no scope's approval-artifact contract (`docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md`) is ever bypassed or synthesized.

## Capability Levels

### Level 0 (existing before this change)
Fully manual: a human runs `system/scripts/business_agent/propose_task.py` for every task, then approves it via the Approval Console, then triggers execution.

### Level 1 (this build — operational)
Event-driven queue population: when a `claude_invocation` or `data_fetch` role's execution succeeds, `system/core/business_agent_trigger.py::evaluate_and_enqueue_next_tasks()` automatically enqueues the role's configured `next_roles` (`system/core/agent_role_registry.py`'s `AgentRoleRecord.next_roles`) as new candidates in `system/runtime/business_agent_tasks/queue.json`, tagged `"source": "auto_trigger"`.

Seed chain (deliberately conservative, not exhaustive fan-out):
- `market_research` → `marketing`
- `marketing` → `doc_creation`
- `analytics` → `pdca`
- `pdca` → `market_research` (Level 2 — see below; terminal at Level 1)

At Level 1, a newly-enqueued task was also activated (made the current shared `agent_handoff/latest.json` scope) *unless* doing so would clobber a different scope that was currently approved and not yet executed. Level 2 (below) removes this shared slot and the guard along with it — every enqueued task is now activated unconditionally, since there is nothing left to clobber. Nothing at either level ever writes `approval.json` or invokes execution.

### Level 2 (operational) — per-task handoff scoping, concurrent chains, still fully human-approved
Goal: multiple businesses' chains — and multiple chains *within* one business (e.g. its PDCA loop and its content chain both progressing) — can progress simultaneously without one clobbering another. See `adr/ADR-0034-business-agent-per-task-handoff-scoping-level2.md`.

1. **Per-task handoff/approval scoping.** Each `(business_id, role_id, task_id)` gets its own handoff/approval files under `system/runtime/agent_handoff/scopes/{business_id}/{role_id}/{task_id}/`, mirroring the existing per-task artifact convention. The top-level `agent_handoff/latest.json`/`approval.json` remain exclusively for the pre-existing issue/draft repo-dev path. `system/core/business_agent_trigger.py`'s anti-clobber guard is deleted, not adjusted — two different scopes never share a path, so there's nothing left to clobber.
2. **File locking.** New `system/core/file_lock.py` (`fcntl.flock`, short timeout, hard error on timeout), applied only to the two files still genuinely shared across all businesses: `business_agent_task_queue.py`'s `queue.json` and `kpi_store.py`'s `kpi.json`. Per-task files need no locking.
3. **Approval Console updated** so each simultaneously-pending `business_role_task` candidate reports its own, independent `approval_status`/`execution_allowed`/`latest_execution_status` — previously all candidates shared one (stale, misleading) read.
4. **Execution concurrency**: `ApprovedAutonomousExecution.run()` and `run_approved_autonomous_execution.py` take optional `business_id`/`role_id`/`task_id` (all-or-nothing), so a human can safely run this in two terminals for two different scopes at the same time. A batch/multi-execute runner (one command running N approved scopes as parallel subprocesses) remains a stretch goal, not built here.
5. **PDCA loop closed**: `pdca → market_research` is now a real `next_roles` edge — safe now that a business's own PDCA-originated proposal and its in-flight content-chain proposal never share a slot.
6. **Release gate, verified**: a cross-scope forgery/leak test (approving scope A's approval must never authorize executing scope B), an issue/draft-path regression test, a two-business execute-independently test, and a same-business PDCA-loop-and-content-chain-coexist test. All pass; see `system/tests/test_agent_execution_approval.py`, `test_approved_autonomous_execution.py`, `test_business_agent_trigger.py`.
7. **Kill switch.** New `system/core/business_agent_automation_control.py` (`pause_automation`/`resume_automation`/`is_automation_paused`, backed by a sentinel file at `system/runtime/business_agent_tasks/automation_paused.json`) and CLI wrappers `system/scripts/business_agent/{pause,resume}_automation.py`. `business_agent_trigger.py::evaluate_and_enqueue_next_tasks()` checks this first and returns immediately (no enqueue, no activation) while paused. Scoped deliberately narrowly: pausing freezes *automatic* next-task proposal only — a human can still manually run `propose_task.py` / `set_execution_approval.py` / `run_approved_autonomous_execution.py` while paused, since an explicit human action should never be blocked by a safety control meant to stop *unattended* chaining. The Approval Console's `render_status()` surfaces an `AUTOMATION PAUSED` banner (reason/who/when) whenever the sentinel is present.

**Known limitations, deliberately not solved here**: no actual parallel/batch execution runner (two terminals, not one command); `queue.json`/`kpi.json` still serialize behind `flock` (rare in practice — human-driven CLI usage, not a hot loop).

### Level 3 — reducing/removing human approval, one lettered sub-stage per boundary crossed
Sub-stages 3a/3b/3c/3d below are each independently built and operational as of this revision (corrected from an earlier draft of this doc that called the whole of Level 3 "future, explicitly not enabled" after 3a/3b/3c already shipped — left uncorrected long enough to become a real internal contradiction, caught during a full Level 1-3 MECE review). What remains genuinely not enabled is any *further* sub-stage beyond 3a-3d (e.g. a persistent OS-level trigger, or extending pre-authorization to `pluggable_provider` roles) — each is forbidden without a new ADR and its own release gate per `docs/current/AUTONOMOUS_OPERATIONAL_BASELINE.md`'s prohibition: *"No new autonomy layer may be enabled unless a new readiness artifact and release gate are added first."* Level 3 as a whole is the one stage that crosses the line `docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md` and `CLAUDE.md` ("Human keeps: strategy, approval, capital allocation") both draw explicitly — treat each sub-stage below as its own deliberate decision that required explicit operator sign-off before its design work started, not something slid into incrementally. One letter names exactly one such crossing; a capability that crosses a distinct boundary gets its own letter rather than being folded into an existing one (see 3d's naming note below, corrected for the same reason as the header above).

- **3a. Policy-based pre-approval of execution (this build — operational, narrowly scoped, no live policy by default).** Built directly on operator request (2026-07-11, two rounds of explicit sign-off, choosing the documented design over broader real-time discretion or a narrower one-off exception). See `adr/ADR-0036-business-agent-level-3a-policy-based-pre-approval.md` for full reasoning.
  - `system/runtime/agent_handoff/auto_approval_policy.json` (hand-authored, PR-reviewed, absent by default — fail closed) authorizes a `(business_id, role_id)` pair: daily/weekly caps, `authored_by`/`authored_at`/`reason`. Validated by `system/core/execution_pre_approval_policy.py`, which re-checks the target role's `role_kind` (`claude_invocation`/`data_fetch` only, never `pluggable_provider`) **and** its `allowed_tools` (no mutating tool) against the live `agent_role_registry` on every single evaluation, not just at authoring time — raising loudly, not silently, if either check fails.
  - `system/core/approved_autonomous_execution.py::ApprovedAutonomousExecution.run()` consults this policy **only** when `evaluate_business_agent_scope_approval()` reports `reason == "missing_approval"` (no `approval.json` exists at all yet) — every other denial reason (rejected, superseded, mismatched, disabled) is completely unaffected, unchanged from Level 0-2. **No code anywhere writes a synthetic `approval.json`** — this is a wholly separate authorization path, and `docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md` needed no amendment.
  - `system/core/execution_pre_approval_state.py`: a per-`(business_id, role_id)`-sharded, locked state machine (`claimed`/`completed`/`failed` per `task_id`) is the sole source of truth for whether a task is in flight, done, or retryable — this closes a real race a first draft had (two near-simultaneous invocations both being told "granted" and both concurrently calling the genuinely-slow `BusinessAgentExecutionAdapter.run()`).
  - A policy-executed task calls `mark_task_status(..., "completed")` itself and persists `authorization_source`/`policy_id` to its runtime execution record and the Approval Console, so it never lingers as a misleading open "candidate."
  - A **third, independent** kill switch (`system/core/execution_pre_approval_control.py`, `pause_pre_approval.py`/`resume_pre_approval.py`) — deliberately **not** the same switch as Level 1/2's `is_automation_paused()`, since that switch's documented contract ("explicit human action stays explicit, never blocked") would otherwise be silently redefined for this new caller.
  - **Known limitations, deliberately not solved here**: per-role caps don't bound aggregate compounding across a `next_roles` chain; composition with Level 3c relies entirely on `finalize_content.py`'s manual step, which no code enforces to involve actual human review (see 3c's note below on why this matters more since 3d shipped); the `missing_approval`-only guarantee holds against every in-repo code path today but isn't filesystem-enforced (an out-of-band `git` operation dropping an `approval.json` would be treated as "missing").
- **3b. Scheduled/unattended execution (built, piloted; persistent OS-level trigger not yet installed).** Built per direct operator request (2026-07-12: "今後も自動化させたい。claudeの手が空かない様にしたい"), scoped explicitly by the operator to generation-only roles — posting/deployment stay human-gated, untouched. See `adr/ADR-0038-business-agent-level-3b-scheduled-unattended-execution.md` for full reasoning.
  - `system/scripts/business_agent/run_scheduled_execution.py` periodically invokes `ApprovedAutonomousExecution.run()` — the exact call a human could make manually — for up to 5 eligible candidates per wake (`system/core/business_agent_task_queue.py::list_candidate_tasks()`, pre-filtered via `get_execution_pre_approval_policy()`, the same function the real authorization check uses, so the pre-filter cannot drift from what's actually authoritative).
  - A **4th, independent** kill switch (`system/core/scheduled_execution_control.py`, `pause_scheduled_execution.py`/`resume_scheduled_execution.py`) — checked first, before the runner even looks at the queue.
  - A whole-invocation lock (`system/core/file_lock.py`) prevents two overlapping wakes from double-executing different scopes if one wake's sequential executions are still running when the next fires.
  - `CLAUDE_EXECUTION_TIMEOUT_SECONDS` is set explicitly in the runner's own process environment (240s) rather than relied on as an inherited shell var — a launchd/cron-invoked process inherits no interactive shell's exported env vars, and this gap already caused a real failure before the fix (`text_syndicate/market_research/task-0003-finance-saas-niche` timed out at the 60s default).
  - Notification: a local, gitignored audit trail (`system/runtime/scheduler/audit/`) every wake, plus an auto-opened PR whenever a wake actually executed something — the operator's own explicit choice after being shown a local-only alternative, since local files alone are invisible until a session reopens. This PR is never merged by 3b itself; see 3d below for what may merge it.
  - **Named cost consequence, stated plainly**: summing every currently-enabled Level 3a policy's `max_auto_approvals_per_day` = 28 real `claude -p` invocations/day, guaranteed, once the scheduler runs often enough to hit every cap daily — a real change from prior sporadic, session-gated usage.
  - **Still not built**: a persistent OS-level trigger (`launchd`/cron) — the runner today is invoked manually or via session-scoped `CronCreate` pilots only. Installing a `launchd` LaunchAgent is a separate, later, explicit operator go-ahead per ADR-0038's release-gate checklist, since it modifies the operator's machine outside this repo.
- **3c. Publishing/distribution automation (this build — operational, publishing only, dry-run only).** Built directly on operator request (2026-07-10, two rounds of explicit sign-off), ahead of 3a/3b for content-generation execution (which remain unbuilt — this does not change how generation gets approved, only what happens to already-approved-and-executed drafts). See `adr/ADR-0035-business-agent-level-3c-policy-based-unattended-publishing.md` for full reasoning.
  - `system/runtime/publishing/policy.json` (hand-authored, PR-reviewed, absent by default — fail closed) authorizes a `(business_id, platform)` pair: allowed source roles (`marketing`/`doc_creation` only), daily/weekly caps, an affiliate-disclosure-tag requirement. Validated by `system/core/publishing_policy.py`.
  - A human runs `system/scripts/publishing/finalize_content.py` after a task has executed successfully, distilling its (often multi-option) raw output into the one exact string that may post. This artifact — not execution approval alone — is the sole eligibility signal; an approved-and-executed task with no finalized content is never published. **This is now the single remaining human touchpoint across the entire generate→land→publish pipeline** for any scope covered by both a 3a policy and 3d's merge gate (see 3d below) — before 3d existed, a human or session at least saw the generation PR's diff before it merged, even without a dedicated review step; that informal secondary look is gone for an all-gates-green 3d merge. No code enforces that this step involves actual human reading of the content, unchanged from before 3d — but the consequence of that pre-existing gap is materially larger now that more of the pipeline around it runs unattended.
  - `system/scripts/publishing/run_scheduled_publish.py` finds all finalized-content candidates, checks policy/caps/dedup/both kill switches/content-integrity (a hash comparison blocks publishing if the underlying execution artifact changed since finalization)/disclosure-tag, and publishes eligible ones via `system/scripts/publishing/providers.py`. **Corrected 2026-07-15**: real X OAuth2 credentials for `text_syndicate` now exist in `.env` (`providers_x.py` is built and live-tested) — this line previously claimed "no platform credentials exist yet for X/Threads/note.com," which is now stale for X specifically. `run_scheduled_publish.py` didn't load `.env` itself (fixed the same day, mirroring `somia/render_content.py`'s existing pattern); a real, isolated single-reply test after that fix reached X's real API but was rejected with a 403 ("not mentioned or otherwise engaged by the author") — an X platform/API-tier policy restriction, not a credential or code problem. Threads/note.com remain uncredentialed.
  - A second, dedicated kill switch (`system/core/publishing_control.py`, `pause_publishing.py`/`resume_publishing.py`) exists alongside the Level 1/2 one; the scheduler stops if *either* is engaged. Neither switch has a manual-override/force path.
  - Publishing state (dedup + counters) is sharded per `(business_id, platform)`, not one shared file, per the same reasoning ADR-0034 already established for per-task state.
  - Audit trail (`system/runtime/publishing/audit/`) surfaced in the Approval Console via a new banner.
  - **Known limitation, deliberately not solved here**: no real OS-level cron/launchd/daemon trigger — the scheduler is a fully-tested, manually-invokable script only. No real analytics/engagement-measurement provider exists for any platform (see 3a's `pdca`/`analytics` note). **No cross-role dedup across a single content thread (e.g. `marketing` and `doc_creation` both finalized for the same platform can both publish)** — worth re-weighing now that 3d makes the generation side of this thread run faster/more unattended than when this limitation was first named.
  - **Content-quality gap found 2026-07-15, addressed separately from this Level ladder**: a DataOps/MarketingOps audit of `text_syndicate`'s finalized-but-unpublished drafts found most were pure engagement bait, and at least one contained fabricated specific facts about products `market_research` never covered (a materially wrong price, invented percentages). See `knowledge/strategy/TEXT_SYNDICATE_CONTENT_STANDARD.md` and the updated `market_research`/`doc_creation` role prompts + `system/scripts/dataops/verify_sourced_facts.py` — this is a content/process-quality fix, orthogonal to the approval-boundary Levels above, but directly relevant to whether 3c should actually go live for a given business.
- **3d. Unattended merge of Level 3b's scheduled-execution PR (this build — operational, gated).** A **distinct boundary crossing from 3b**, not a sub-feature of it: 3b schedules generation; 3d decides whether 3b's own opened PR may land on `main` with zero human/session review. Built directly on operator request, after this session hit the real cost of the pre-3d status quo (an unmerged 3b PR caused two subsequent wakes to re-execute the same 3 already-succeeded scopes, burning real `claude -p` calls for nothing) — "構いません。claudeを信頼します。ただし、claudeがその信頼に応えるべき自己チェック体制を備えてください." See `adr/ADR-0040-business-agent-level-3d-unattended-merge.md` for full reasoning, including its own naming-correction note (originally filed as a 3b sub-feature; relabeled 3d after this same MECE review).
  - Merges only if **10 independent, fail-closed gates** all pass: every scope in the wake succeeded; the committed diff re-verified against the runtime-path allowlist; local `pytest`/`validate_all.py` clean; a stale-base check against GitHub's own remote state immediately before merging (the direct fix for a real collision this session hit: two unmerged 3b PRs re-executing the same work); GitHub's real CI polled green with a bounded timeout; a **5th, dedicated** kill switch (`system/core/scheduled_merge_control.py`); a circuit breaker that auto-pauses that switch after 3 consecutive gate failures of any kind; `business_id`/`role_id`/`task_id` whitelist-validated before a wake even attempts execution; every attempt/skip recorded in the existing audit trail; and `system/scripts/business_agent/rollback_scheduled_merge.py`, which prepares a revert PR for a bad merge but never merges it itself. Any single failed gate leaves the PR open for human/session review, exactly as 3b alone behaved before 3d existed.
  - **Known limitation, deliberately not solved here**: no automated post-merge content review — a bad merge still requires a human or session to *notice* it before `rollback_scheduled_merge.py` is useful. See 3c's finalize_content.py note above for how this compounds with publishing.

**Adjacent, orthogonal concern — not part of this Level ladder**: `adr/ADR-0039-agent-role-definition-dual-authority.md` documents that the 8 business-content roles this entire ladder governs execution for are defined **twice** — `system/agent_runtime/role_prompts/*.md` (authoritative for every automated path above: 3a/3b/3d) and `.claude/agents/*.md` (authoritative for interactive Claude Code sessions only). Neither this document nor `CLAUDE.md`/`AGENTS.md` cross-referenced that arrangement until this note — a reader of this ladder alone would not know a role's automated behavior and its interactive-session behavior are governed by two separately-maintained files that must be updated together when they diverge on substance.

**Recommended sequencing**: ~~Level 2 (parallel, still human-approved)~~ done → ~~Level 3c (publishing, dry-run only)~~ done, ahead of sequence by explicit operator request → ~~Level 3a (policy pre-approval of execution)~~ done → ~~Level 3b (scheduled/unattended execution under that policy)~~ done and piloted (2026-07-12) → ~~Level 3d (unattended merge of 3b's PR)~~ done, gated, end-to-end verified for real (2026-07-14/15). **Next, not yet decided**: a persistent `launchd` trigger for 3b, gated by ADR-0038's release-gate checklist — deliberately not bundled with 3d, since installing something that runs on the operator's machine independent of any session is a materially different commitment than a gated merge capability.

## Verified Example
- Level 1: `text_syndicate/market_research/task-0001` (real, evidence-grounded output) → auto-enqueued and activated `text_syndicate/marketing/auto-0001`, which itself ran for real and produced a genuine marketing draft.
- Level 2: ran the trigger for real against production data — `text_syndicate/pdca` and `text_syndicate/market_research`'s content chain both had independently-pending, independently-tracked scopes for the same business at once, with no interference (see `test_business_agent_trigger.py::test_pdca_loop_and_content_chain_coexist_for_same_business`); a cross-scope forgery test confirms approving one business's scope can never authorize executing a different one.
- Kill switch: ran `pause_automation.py` against production, confirmed the console shows the `AUTOMATION PAUSED` banner and the trigger returns `[]` untouched, then ran `propose_task.py` manually while still paused and confirmed it still worked, then `resume_automation.py` and confirmed clean resumption with no trace left behind.
- Command sequence: unchanged from Level 0/1 for a single scope — `propose_task.py` (or an auto-triggered enqueue) → `set_execution_approval.py` (human) → `run_approved_autonomous_execution.py`, now with optional `--business-id`/`--role-id`/`--task-id` to target a specific scope.
- Level 3a: authored a throwaway pilot policy entry (`max_auto_approvals_per_day: 1`) for a real, missing-approval `text_syndicate` scope, ran `run_approved_autonomous_execution.py` with no `approval.json` present, confirmed `authorization_source=policy_pre_approval` and a real execution artifact, confirmed an immediate re-run reported the skipped-duplicate path (not a second real execution), confirmed a second distinct task under the same scope was denied on the cap, confirmed `pause_pre_approval.py`/`resume_pre_approval.py` work independently of `pause_automation.py`, and confirmed the queue entry moved to `"completed"` so it no longer appeared as an open candidate in the Approval Console. The throwaway policy entry and demonstration artifacts were removed before opening the PR — no live policy ships by default.
- Level 3d: ran a real scheduled wake against the live backlog; 3 scopes executed successfully, all 10 gates passed (including a real stale-base check against GitHub's actual `origin/main`), and the PR merged fully unattended — content spot-checked as genuine, not a false success. Separately confirmed the circuit breaker trips for real after 3 consecutive induced gate failures (`paused_by="circuit_breaker"`) and clears via `resume_scheduled_merge.py`. Separately confirmed `rollback_scheduled_merge.py` against a real, deliberately-bad, clearly-labeled throwaway merge: it opened a correct revert PR and did not auto-merge it, reviewed and merged manually with no permanent trace left.

## Forbidden Now

Each item below names the real concern it protects (reversibility/audit,
cost/blast-radius, or the approval contract itself), not just a bare
technology ban — see `adr/ADR-0037` for why this matters: a rule stated as
an unqualified absolute tends to go stale and silently contradict a later
real decision instead of being revisited when circumstances change.

- **No real OS-level cron/launchd entry or persistent daemon installed anywhere, for anything** (including the Level 3c publishing scheduler and the Level 3b scheduler — all are manually- or `CronCreate`-session-invoked scripts only, per ADR-0038). *Concern: reversibility/audit.* A persistent, session-independent trigger is a separate, later, explicit ask gated by ADR-0038's release-gate checklist — it would run outside this repo's git-tracked safety net (see the closing paragraph below) with no active session present to notice something going wrong in real time.
- **No GitHub Actions or other CI wiring into this flow.** *Concern: reversibility/audit* — same reasoning as the cron/daemon ban above: a CI-triggered execution is another unattended-trigger path lacking a human checkpoint at the moment it runs.
- **No removing or bypassing the human approval gate itself** (`docs/current/AUTONOMOUS_EXECUTION_APPROVAL_CONTRACT.md`, `approval.json`'s `decision_status: approved`). *Concern: this is the one absolute in this list that really is permanent, not a stand-in for something narrower* — matches `CLAUDE.md`'s "Human keeps: strategy, approval, capital allocation." That field stays exclusively human-written for every scope, forever; Level 3a only adds a separate, additional authorization path consulted for scopes with no approval decision at all, and Level 3c only acts on drafts a human already separately approved-and-executed through that gate.
- **No real publishing provider (X/Threads/note.com) goes live before the operator independently obtains and wires real credentials** — `dry_run` is the only provider today. *Concern: irreversibility (a real post to a real platform can't be undone) + cost (real credentials imply a real account/spend commitment).*
- **No generalizing Level 3a's or Level 3c's policy-based pre-authorization to `pluggable_provider` roles (image/video generation), or to any role whose `allowed_tools` include a mutating tool** — untouched, still forbidden. *Concern: cost/runaway-spend*, stated explicitly in this doc's own Level 3b section below ("a runaway loop costs real money, not just tokens"). If a future `pluggable_provider` role were free or hard-capped, this line should be revisited rather than assumed to still apply by default — the ban tracks the cost risk, not the role category for its own sake.
- **No GitHub mutation beyond the existing bounded approved flow, with two explicit, gated carve-outs**: Level 3b may open a PR when a wake actually executes something (ADR-0038), and Level 3d may merge that PR unattended if every one of its 10 independent gates passes -- refusing to touch a working tree that wasn't already clean before the wake started, in both cases. *Concern: blast-radius/irreversibility* of repo-state mutation outside the reviewed path — bounded by fail-closed gates (any single gate failure leaves the PR open for human/session review, the pre-3d behavior) rather than by human review being unconditionally required before every landing.

**Runtime state here is git-tracked**, and this is itself a second, independent safety backstop beyond the approval gate at every level. Before Level 3d, a successful auto-trigger chain link only became durable/visible on `main` once a human or session reviewed and merged its PR. As of 3d, an all-gates-green wake's PR may merge unattended -- the git-tracked history (and `rollback_scheduled_merge.py`) is the backstop that remains, not a guaranteed pre-merge human review.
